My Journey as a Web Developer: From Curious Student to Craftsman

By David John Emmanuel G. Tulio

The Beginning: View Source

My journey into web development began not with a formal class or a textbook, but with a simple act of curiosity. I was in high school, browsing a website I admired, when I right-clicked and selected "View Page Source." The screen filled with what looked like alien code — tags, attributes, angle brackets everywhere. Instead of closing it, I stayed.

I started Googling what each tag meant. h1 for headings. p for paragraphs. a for links. Within an hour, I had built my first webpage — just a white page with a heading and a paragraph — but I was hooked. It felt like discovering a superpower: I could build things that anyone in the world could see.

The Learning Curve

CSS was humbling. Centering a div felt like solving a puzzle with missing pieces. JavaScript was even harder — I spent an entire weekend debugging a function that was failing simply because of a missing semicolon. But every breakthrough felt euphoric, and the cycle of confusion → research → understanding → build became addictive.

I committed to building something every week. Small projects at first — a to-do list, a weather app, a simple landing page for a fictional business. Each project taught me something the tutorials didn't: how to make decisions when there's no right answer.

Key Lessons I've Learned

  • Build real things as early as possible — tutorials can only take you so far
  • Embrace being stuck; it's where the deepest learning happens
  • Read other people's code — it expands your vocabulary of solutions
  • Accessibility and performance are not optional features, they are foundations
  • The web is a community — give back by sharing what you learn

Where I Am Now

Today I build full-stack applications with React, Node.js, and Python. I care deeply about the details — the transition timing on a hover state, the semantic structure of an HTML document, the milliseconds shaved off a page load. Web development taught me that craft lives in the details.

If you're just starting out: keep going. The confusion you feel right now is not a sign that you're not cut out for this — it's a sign that you're learning.

Tips for Building Responsive Websites That Actually Work

By David John Emmanuel G. Tulio

Start with Mobile First

The single most effective shift in my responsive workflow was committing to mobile-first CSS. Instead of writing desktop styles and then overriding them with media queries, I write the base styles for small screens and progressively enhance for larger ones. The result is leaner CSS and a layout that degrades gracefully — not one that tries to squeeze a desktop layout into a phone.

Use Fluid Typography

Fixed font sizes break at the edges of your breakpoints. The CSS clamp() function changed everything for me. Instead of setting five breakpoints for a heading, I write one line: font-size: clamp(1.8rem, 5vw, 4rem); The font grows smoothly between a minimum and maximum size, and the browser handles the math.

Practical Tips That Made the Biggest Difference

  • Use min(), max(), and clamp() for fluid spacing
  • CSS Grid with auto-fit and minmax() replaces most media queries for card layouts
  • Test on real devices, not just browser DevTools — they behave differently
  • Use aspect-ratio to maintain proportions without fixed heights
  • Avoid vh for mobile — use svh (small viewport height) instead
  • Set overflow-x: hidden on the body only when you understand why

The Real Secret

The best responsive websites aren't built by adding breakpoints — they're built by designing with fluid, intrinsic layouts from the start. When your base layout is flexible, you need far fewer media queries. Think in proportions, not pixels, and responsive design becomes natural.

Web Performance in 2024: What Actually Moves the Needle

By David John Emmanuel G. Tulio

The Problem With Generic Performance Advice

Every web performance article tells you the same things — compress your images, use a CDN, minify your CSS. That advice is not wrong, but it rarely tells you which optimizations will actually make a visible difference to real users. After working on several projects and studying Core Web Vitals, here is what I have found genuinely moves the needle.

Core Web Vitals: Focus on LCP First

Google's Core Web Vitals — LCP, CLS, and INP — are now ranking signals. Of the three, Largest Contentful Paint (LCP) has the most direct impact on perceived load speed. LCP measures how long it takes for the largest visible element (usually a hero image or heading) to render. The single most effective fix is adding fetchpriority="high" to your above-the-fold hero image. This tells the browser to load it before anything else, often cutting LCP by 30–50%.

Lazy Loading Is Not a Silver Bullet

Adding loading="lazy" to every image is a common mistake. Lazy loading delays images that are off-screen — but if you apply it to images that are immediately visible, you are actually slowing down your page. The rule is simple: only lazy load images that are below the fold. The hero image and any image in the first viewport should never be lazy loaded.

What Actually Made the Biggest Difference in My Projects

  • Converting images from PNG/JPG to WebP — typically 25–35% smaller files with no visible quality loss
  • Adding width and height attributes to every image — prevents layout shift (CLS) as images load
  • Using font-display: swap on custom fonts — prevents invisible text while fonts load
  • Deferring non-critical JavaScript with defer or async — speeds up initial render
  • Removing unused CSS — tools like PurgeCSS can cut stylesheet size dramatically

The Honest Truth About Bundle Optimization

If you are building a plain HTML/CSS/JS site without a framework, bundle optimization is largely irrelevant. The biggest wins come from the basics — optimized images, deferred scripts, and efficient CSS. Complex build pipelines are tools for solving complex problems. For simpler projects, keep it simple and measure what actually matters with Lighthouse and PageSpeed Insights before optimizing anything.
